# RoPE: Rotary Position Embedding

## Citation

Su, J., Lu, Y., Pan, S., Murtadha, A., Wen, B., & Liu, Y. (2021).
**RoFormer: Enhanced Transformer with Rotary Position Embedding.**
arXiv:2104.09864 [cs.CL].
https://arxiv.org/abs/2104.09864

## Abstract

The paper investigates methods for integrating positional information into
transformer-based language models and proposes Rotary Position Embedding (RoPE).
RoPE "encodes the absolute position with a rotation matrix and meanwhile
incorporates the explicit relative position dependency in self-attention
formulation." It offers "flexibility of sequence length, decaying inter-token
dependency with increasing relative distances." The resulting model, RoFormer,
demonstrates consistent improvements over alternatives on long text classification
benchmarks. RoFormer is integrated into HuggingFace.

## Key Equations

### Inverse Frequency (theta) Formula

For a model of dimension d, the per-dimension-pair frequency is:

```
θ_j = 10000^(-2j/d),  j = 0, 1, ..., d/2 - 1
```

Equivalently, the inverse frequencies vector:

```
inv_freq[j] = 1 / (10000 ^ (2j / d))
```

This gives a geometric sequence of frequencies, with lower dimensions rotating
faster and higher dimensions rotating more slowly — analogous to sinusoidal
position encodings but applied via rotation.

### Position-Dependent Transformation

For a token at position m with query vector q (or key vector k) of dimension d,
the rotated version is produced by applying a block-diagonal rotation matrix R:

```
f_q(q, m) = R_{θ,m} · q
f_k(k, n) = R_{θ,n} · k
```

The rotation matrix R_{θ,m} is block-diagonal with 2×2 rotation blocks:

```
R_{θ,m} = diag(
    [cos(m·θ_0), -sin(m·θ_0)],  sin(m·θ_0),  cos(m·θ_0)],
    [cos(m·θ_1), -sin(m·θ_1)],  sin(m·θ_1),  cos(m·θ_1)],
    ...
    [cos(m·θ_{d/2-1}), -sin(m·θ_{d/2-1})],  ...
)
```

Applied element-wise, for consecutive pairs (x_{2j}, x_{2j+1}):

```
[x_{2j}']   = [cos(m·θ_j)  -sin(m·θ_j)] [x_{2j}  ]
[x_{2j+1}']   [sin(m·θ_j)   cos(m·θ_j)] [x_{2j+1}]
```

### The rotate_half Form (equivalent implementation)

In practice the rotation is computed as:

```
rotate_half(x):  splits x into [x1 | x2], returns [-x2 | x1]

rotated(x, m) = x * cos(m·Θ) + rotate_half(x) * sin(m·Θ)
```

where cos(m·Θ) and sin(m·Θ) are vectors of cos/sin values for each dimension
pair, broadcast over the full d-dimensional vector.

### Relative Position Property (Key Correctness Criterion)

The inner product of the rotated query and key depends ONLY on their relative
position (m - n), not on absolute positions:

```
⟨f_q(q, m), f_k(k, n)⟩ = g(q, k, m - n)
```

This follows because:

```
R_{θ,m}^T · R_{θ,n} = R_{θ,m-n}
```

i.e., the product of two rotation matrices at positions m and n is a rotation
matrix at the relative offset m-n.  The attention score becomes:

```
score(m, n) = (R_{θ,m} q)^T (R_{θ,n} k)
            = q^T R_{θ,m-n} k
```

This is the core property that gives RoPE its relative-position semantics while
only requiring per-token rotation (O(d) per token, not O(d²)).

### Arbitrary Sequence Length

RoPE supports arbitrary sequence lengths without modification: the rotation is
applied position-by-position up to whatever length is present. No learned
embeddings are involved; positions outside the training range simply use the
same trigonometric formula extrapolated. In practice, models may lose some
precision for very long out-of-distribution contexts, but the formulation itself
has no hard length limit.

## Implementation Notes for Verification

A correct RoPE implementation must satisfy:

1. **Frequencies**: `inv_freq = 1.0 / (base ** (torch.arange(0, dim, 2).float() / dim))`
   with `base = 10000` by default.

2. **Positional grid**: `freqs = outer(positions, inv_freq)`; then
   `cos_table = cos(freqs)`, `sin_table = sin(freqs)`.

3. **Application**: `x_rot = x * cos + rotate_half(x) * sin`
   where `rotate_half` splits the last dim in half and returns `[-x2, x1]`.

4. **Applied to both q and k** with the same cos/sin tables indexed by position.

5. **Relative position test**: `dot(rotate(q, m), rotate(k, n))` should equal
   `dot(rotate(q, 0), rotate(k, n-m))` for all valid m, n.

6. **Complex-number equivalence**: the rotate_half form is equivalent to
   treating pairs as complex numbers and multiplying by `exp(i·m·θ_j)`.
   The LLaMA implementation uses `torch.view_as_complex` to exploit this directly.
