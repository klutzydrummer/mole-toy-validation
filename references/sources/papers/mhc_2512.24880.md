## Abstract (verbatim)

> "Recently, studies exemplified by Hyper-Connections (HC) have extended the ubiquitous
> residual connection paradigm established over the past decade by expanding the residual
> stream width and diversifying connectivity patterns. While yielding substantial performance
> gains, this diversification fundamentally compromises the identity mapping property
> intrinsic to the residual connection, which causes severe training instability and
> restricted scalability, and additionally incurs notable memory access overhead. To address
> these challenges, we propose Manifold-Constrained Hyper-Connections (mHC), a general
> framework that projects the residual connection space of HC onto a specific manifold to
> restore the identity mapping property, while incorporating rigorous infrastructure
> optimization to ensure efficiency. Empirical experiments demonstrate that mHC is effective
> for training at scale, offering tangible performance improvements and superior scalability.
> We anticipate that mHC, as a flexible and practical extension of HC, will contribute to a
> deeper understanding of topological architecture design and suggest promising directions
> for the evolution of foundational models."

## Section 3: Preliminary

The l-th layer input **x**_l ∈ ℝ^(1×C) is expanded by factor n to create a hidden matrix
**x̄**_l ∈ ℝ^(n×C) representing n parallel residual streams.

Three learnable mappings govern the layer:
- **ℋ^res_l** ∈ ℝ^(n×n) — residual stream mixing matrix
- **ℋ^pre_l** ∈ ℝ^(1×n) — maps n streams to sublayer input
- **ℋ^post_l** ∈ ℝ^(1×n) — distributes sublayer output back to n streams

**Standard residual connection (Eq. 1):**

    x_{l+1} = x_l + F(x_l, W_l)

**HC multi-layer propagation (Eq. 2):**

    x_L = x_l + Σ_{i=l}^{L-1} F(x_i, W_i)

## Section 4: Method

### Core mHC Formula (Eq. 3)

    x_{l+1} = H^res_l · x_l  +  H^post_l^T · F(H^pre_l · x_l,  W_l)

Multi-layer unrolled (Eq. 4):

    x_L = (∏_{i=1}^{L-l} H^res_{L-i}) · x_l
          + Σ_{i=l}^{L-1} (∏_{j=1}^{L-1-i} H^res_{L-j}) · H^post_i^T · F(H^pre_i · x_i, W_i)

---

### Section 4.1: Manifold Constraint — Doubly Stochastic Property (Eq. 6)

The residual matrix **ℋ^res_l** is constrained to the Birkhoff polytope (doubly stochastic
matrices):

    P_M^res(H^res_l) := { H^res_l ∈ ℝ^(n×n) | H^res_l · 1_n = 1_n,
                                                  1_n^T · H^res_l = 1_n^T,
                                                  H^res_l ≥ 0 }

This enforces:
1. **Row sums = 1, column sums = 1, all elements ≥ 0**
2. **Spectral norm bounded ≤ 1** — prevents gradient explosion
3. **Closed under matrix multiplication** — the product of doubly stochastic matrices is
   doubly stochastic, so stability is preserved across depth
4. **Geometric interpretation** — Birkhoff polytope is the convex hull of all permutation
   matrices; "the set of doubly stochastic matrices is closed under matrix multiplication...
   the composite residual mapping across multiple layers remains doubly stochastic, thereby
   preserving stability throughout the entire depth."

**Why not softplus?** Softplus (or any elementwise positive-only activation) does NOT
enforce row-sum = 1 or column-sum = 1, so it cannot project onto the doubly stochastic
manifold. Sinkhorn-Knopp is required for the doubly-stochastic projection.

**Why sigmoid for H_pre / H_post (not softmax, not softplus)?**
- Non-negativity is required (sigmoid ensures ≥ 0) to prevent signal cancellation
- Softmax would enforce Σ = 1 exactly; sigmoid does not (more flexibility)
- The paper uses sigmoid specifically: ℋ^pre_l = σ(H̃^pre_l), ℋ^post_l = 2σ(H̃^post_l)
- The 2× factor for H_post constrains outputs to [0, 2] range for stability

---

### Section 4.2: Parameterization and Projection (Equations 7–9)

**Parameterization (Eq. 7 — for H_res):**

    x̃_l = RMSNorm(x_l)
    H̃^res_l = α^res_l · mat(x̃_l · φ^res_l) + b^res_l

where α^res_l is a learnable scalar gating factor initialized to 0.01.

**Parameterization (Eq. 8 — for H_pre, H_post):**

    H^pre_l  = σ(H̃^pre_l)
    H^post_l = 2σ(H̃^post_l)

σ = sigmoid function.

**Sinkhorn-Knopp Projection (Eq. 9):**

    M^(0) = exp(H̃^res_l)
    M^(t) = T_r(T_c(M^(t-1)))

where T_r and T_c denote row and column normalization respectively.
**t_max = 20 iterations** used in practice.

---

### Section 4.3: Infrastructure

- **Kernel Fusion:** Mixed-precision kernels (TileLang) fuse RMSNorm, matrix multiplications,
  and Sinkhorn iterations to reduce memory bandwidth overhead.
- **Recomputing (gradient checkpointing):** Intermediate mHC activations are discarded
  after the forward pass and recomputed during backpropagation. Optimal block size:
  L*_r ≈ √(nL / (n+2))
- **DualPipe Extension:** ℱ_post,res kernels execute on dedicated compute streams to
  overlap communication at pipeline stage boundaries.

## Implementation Notes vs. Reference Code

The reference code at `/home/brandon/Projects/MoLE/reference/mhc_hyper_connections.py`
(sourced from `tokenbender/mHC-manifold-constrained-hyper-connections`) differs from the
paper in several ways:

| Aspect | Paper (arXiv 2512.24880) | Reference Code |
|--------|--------------------------|----------------|
| Sinkhorn domain | Standard (exp then normalize) | **Log-space** (logsumexp) |
| Temperature τ | Not mentioned | τ = 0.05 |
| Sinkhorn iterations | t_max = 20 | num_iters = 10 default |
| H_res activation | Sinkhorn-Knopp | `sinkhorn_log()` |
| H_pre activation | σ (sigmoid) | **softmax** (`.softmax(dim=-1)`) |
| H_post activation | 2σ (2×sigmoid) | **softmax** (`.softmax(dim=-1)`) |
| H_res_logits init | "α=0.01 gating" | `full(-8.0)`, diagonal set to `0.0` |
| H_pre_logits init | Not specified | `full(-8.0)`, one entry per view set to `0.0` |
| H_post_logits init | Not specified | `zeros(1, num_streams)` |

**Key discrepancy — softmax vs sigmoid:**
The paper (Eq. 8) uses sigmoid (σ) for H_pre and 2σ for H_post. The reference code uses
`softmax(dim=-1)` for both. The reference code's comment header says "softmax NOT softplus"
and the comment explicitly notes softmax is what the implementation uses. The paper's
sigmoid formulation and the code's softmax are functionally different (softmax enforces
Σ=1; sigmoid does not). For the MoLE toy-validation implementation, the **code** should be
treated as authoritative since it is the released implementation.

**Log-space Sinkhorn:**
The paper describes standard Sinkhorn (exp + normalize). The reference code implements a
numerically stable log-space variant with temperature τ=0.05, which is more stable for
small logit values. This is the actual implementation used.

## sinkhorn_log — Verbatim from Reference Code

```python
def sinkhorn_log(logits, num_iters=10, tau=0.05):
    n = logits.shape[-1]
    Z = logits / tau
    log_marginal = torch.zeros((n,), device=logits.device, dtype=logits.dtype)

    u = torch.zeros(logits.shape[:-1], device=Z.device, dtype=Z.dtype)
    v = torch.zeros_like(u)

    for _ in range(num_iters):
        u = log_marginal - torch.logsumexp(Z + v.unsqueeze(-2), dim=-1)
        v = log_marginal - torch.logsumexp(Z + u.unsqueeze(-1), dim=-2)

    return torch.exp(Z + u.unsqueeze(-1) + v.unsqueeze(-2))
```

**Notes on sinkhorn_log:**
- `tau = 0.05` is the temperature (scales logits before Sinkhorn)
- `num_iters = 10` default (paper says 20; lite paper arXiv:2601.05732 argues fewer suffice)
- Handles both `[n, n]` inputs (plain matrix) and `[B, L, n, n]` inputs (batched) because
  `u` and `v` are shaped to `logits.shape[:-1]`, so broadcasting works for any leading dims
- `log_marginal = zeros(n)` corresponds to target log-marginals of log(1/n)·n = 0,
  i.e., uniform marginals (doubly stochastic target)
- Returns `exp(Z + u + v)` which is the doubly stochastic matrix in probability space

## H Matrix Initializations — Verbatim from Reference Code

```python
# H_res: near-identity doubly stochastic start
init_h_res = torch.full((num_residual_streams, num_residual_streams), -8.0)
init_h_res.fill_diagonal_(0.0)
self.H_res_logits = nn.Parameter(init_h_res)
# After sinkhorn_log: diagonal ≈ 1.0, off-diagonal ≈ 0.0  (near identity)

# H_pre: near-one-hot start (picks one stream per view)
init_h_pre = torch.full((num_input_views, num_residual_streams), -8.0)
init_h_pre[:, init_residual_index] = 0.0
self.H_pre_logits = nn.Parameter(init_h_pre)
# After softmax: one entry ≈ 1.0, rest ≈ 0.0  (hard selection of one stream)

# H_post: uniform start
self.H_post_logits = nn.Parameter(
    torch.zeros(num_input_views, num_residual_streams)
)
# After softmax: all entries = 1/num_streams  (uniform distribution to all streams)
```
