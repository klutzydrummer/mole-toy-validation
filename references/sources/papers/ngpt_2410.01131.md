## Abstract (verbatim)

> "We propose nGPT (normalized GPT), a new transformer architecture where all vectors that
> form the residual stream are constrained to lie on a unit hypersphere. This includes
> embeddings, MLP, attention matrices and hidden states. The primary result is that the input
> stream of tokens travels on the surface of a hypersphere, with each layer contributing
> a displacement towards the target output predictions. Using this framework, we demonstrate
> that nGPT learns 4 to 20 times faster than the original GPT for context lengths of 1k–8k
> tokens."

## Core innovation: hyperspherical update rule

Standard transformer residual:
```
h ← h + F(RMSNorm(h))
```

nGPT spherical update (Eq. 10–11):
```
h_sub = L2Norm(F(h))              # sublayer output normalized to sphere
h     = L2Norm(h + α ⊙ (h_sub − h))   # LERP interpolation then renorm
```

- `α ∈ ℝ^d` — per-dim learnable "eigen learning rate", init ≈ 0.05/√d
- `L2Norm(x) = x / ||x||₂` — normalizes along last dim
- The paper frames each layer as an optimization step on the unit hypersphere
- Connects to SLERP (spherical linear interpolation) approximated by LERP

## Attention modifications

**Q and K normalization:**
```
q = L2Norm(q)    # per head, after RoPE
k = L2Norm(k)    # per head, after RoPE
```
Dot products become cosine similarities in [−1, 1].

**Inverted attention scale:**
```
attention = softmax(q @ k.T * s_qk / sqrt(d_h)) @ v
```
But with normalized Q/K, the paper uses `s_qk` (learnable, init √d_h) and `scale=1.0`:
```
attention = softmax(q * s_qk @ k.T) @ v    # effectively scale=s_qk, not 1/√d_h
```
`s_qk` is a learnable scalar (init √d_h ≈ 8.0 at d=512, n_heads=8), applied as `q * s_qk`
before SDPA with `scale=1.0` so gradient flows through `s_qk`.

**MLP scaling factors:**
`s_u`, `s_v` learnable scalars for intermediate MLP states (compensate for normalized activations).

**Logit scale:**
`s_z` learnable scalar (init √d ≈ 22.6) multiplies final logits (embed.weight is unit-norm
→ logits are cosine similarities; s_z restores softmax temperature).

## Weight normalization (post-optimizer-step)

After each `optimizer.step()`:
```python
@torch.no_grad()
def normalize_weights(model):
    for param_matrix in all_weight_matrices(model):
        param_matrix.data = F.normalize(param_matrix.data, dim=-1)  # row normalization
```

Covers: embed, lm_head, all Linear weight matrices in attention and FFN.
Ensures all weight row-vectors are unit-norm, consistent with normalized hidden states.
Paper calls this "representation learning on the hypersphere."

## Alpha parameters

- `α_a` (attention), `α_m` (MLP): per-dim learnable, shape [d]
- Init: 0.05 (paper), with effective step = `0.05 / √d ≈ 0.0022` at d=512
- Scale factor `1/√d` applied at forward time (not stored): `alpha_eff = α * (1/√d)`
- Networks empirically learn 20–30% step sizes during training (much larger than init)

---

## What becomes redundant (kept for compatibility)

- `norm1`, `norm2` pre-norms: input is already unit-norm → RMSNorm becomes scalar
- `norm_out` final norm: identity on unit-norm input
- Weight decay: explicitly removed — conflicts with per-step weight normalization
- Learning rate warmup: paper suggests removing it, but retaining warmup is safe

## Convergence speedup results

Experiments on **OpenWebText dataset**, **0.5B and 1B parameter models**:

| Context length | Speedup |
|----------------|---------|
| 1k tokens      | ~4×     |
| 4k tokens      | ~10×    |
| 8k tokens      | ~20×    |

nGPT reaches GPT's 200k-iteration validation loss at only 20k iterations (4k context).
Longer context benefits more — authors attribute this to better representation geometry.

**Caveat (from NVIDIA README):** The official repo uses bfloat16 throughout, while the
baseline nanoGPT uses float32. This moderately affects nGPT but greatly degrades the GPT
baseline, artificially inflating measured speedups. Actual speedups are smaller than 4–20×.

---

## Length extrapolation

- nGPT exhibits superior length extrapolation on PG19 (tested up to 32k tokens)
- Stable perplexity beyond pre-training length without modifying RoPE
- Authors attribute this to uniform hyperspherical representation geometry

---

## Computational overhead

- 60–80% per-step overhead due to unoptimized normalization ops
- Overhead diminishes for larger models
- Weight normalization is O(parameters) and adds a full model pass per step

## Reference implementation notes (from https://github.com/NVIDIA/ngpt)

Key function (verbatim from repo):
```python
def justnorm(self, x):
    res = x / x.norm(p=2, dim=-1, keepdim=True)
    return res
```

Alpha-based update (verbatim from repo):
```python
lr = self.attn_alpha * (self.attn_alpha_init_value / self.attn_alpha_init_scaling)
lr = torch.abs(lr)
A_norm = self.justnorm(h)
B_norm = self.justnorm(h_att)
res = A_norm + lr * (B_norm - A_norm)
h = self.justnorm(res)
```

Note: the repo normalizes `h` (pre-update) and `h_att` (sublayer output) separately
before the LERP, then renormalizes. This matches Eq. 10–11 exactly.
